# PR 코드리뷰 스킬

호출 시 아래 순서로 실행합니다.

## 실행 순서

1. 현재 브랜치와 main 브랜치의 diff를 확인합니다.
   ```bash
   git diff main...HEAD
   ```

2. 변경된 Java · Kotlin · YAML · Gradle 파일을 직접 읽어 맥락을 파악합니다.

3. 아래 리뷰 포맷으로 한국어 코드리뷰를 작성합니다.

4. 열린 PR 번호를 확인하고 코멘트를 게시합니다.
   ```bash
   gh pr view --json number
   gh pr comment <number> --body "<리뷰 내용>"
   ```

## 리뷰 포맷

```
## ✅ 잘된 점
잘 작성된 부분을 구체적으로 언급합니다.

## 🔧 개선 제안
- **[P1] Blocker**: 머지 전 반드시 수정 — 보안 취약점, 데이터 손실 위험
- **[P2] Critical**: 머지 전 강하게 권장 — 테스트 미커버, 회귀 위험
- **[P3] Major**: 후속 PR 수정 — 성능·가독성에 큰 영향
- **[P4] Minor**: 선택 사항 — 변수명, 주석, 작은 리팩토링
- **[P5] Nit**: 단순 의견 — 오타, 여백

## ❓ 질문
이해가 필요한 부분을 "어떤 의도였나요?" 형식으로 질문합니다.
```

## 규칙

- 존댓말로 작성합니다.
- 비판이 아닌 제안 형식으로 작성합니다.
- 변경이 없는 영역은 언급하지 않습니다.
- PR이 열려 있지 않으면 리뷰 내용만 출력하고 코멘트 게시는 건너뜁니다.
- 개선 제안이 없으면 해당 섹션을 생략합니다.
