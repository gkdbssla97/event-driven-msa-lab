# 로컬 k3s 운영 메모

이 저장소는 **애플리케이션 코드와 클러스터 배포 자산을 함께 관리**하지만, `k3s` 바이너리 자체를 설치하거나 머신 전체를 초기화하지는 않습니다.

즉, 이 저장소의 `scripts/dev/*.sh` 는 아래 전제를 둡니다.

- `kubectl` 이 설치되어 있다.
- `helm` 이 설치되어 있다.
- 현재 `kubectl` 컨텍스트가 **로컬 k3s 클러스터**를 가리킨다.

## 왜 이렇게 나눴는가

`k3s` 설치 방식은 운영체제와 로컬 개발 환경에 따라 달라집니다. 특히 macOS 에서는 Linux 서버처럼 `sudo k3s server` 로 바로 올리는 방식보다 별도 VM/런타임 전략이 섞이기 쉽습니다.

그래서 이 저장소는 다음 범위까지만 책임집니다.

1. k3s 컨텍스트가 준비된 뒤
2. Kafka 를 Helm 으로 설치하고
3. 상태를 확인하고
4. 토픽을 다루는 스크립트를 제공한다.

## 권장 흐름

1. 로컬에서 k3s 클러스터를 준비한다.
2. `kubectl config current-context` 로 대상 컨텍스트를 확인한다.
3. 저장소 루트에서 `make dev-up` 을 실행한다.
4. `make dev-status` 로 Kafka 관련 리소스를 확인한다.

## 주의 사항

- 현재 기본 Kafka 배포는 **클러스터 내부 접근** 기준입니다.
- 즉, Spring Boot 서비스를 호스트에서 직접 띄워 클러스터 Kafka 에 접속하려면 이후 단계에서 external listener 전략을 추가해야 합니다.
- 지금 단계의 목적은 **로컬 k3s 에 Kafka 를 일관되게 올리는 저장소 내 운영 자산을 만드는 것**입니다.
